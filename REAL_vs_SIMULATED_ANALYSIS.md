## 为什么真实数据效果更好？

### 1. 几何参数准确性 ⭐️

**真实数据 (AAPM)**:
- 来自真实临床CT扫描仪
- 几何参数（SOD, SDD, 角度）由扫描仪硬件精确控制
- DICOM标签由扫描仪固件直接写入，准确可靠
- 角度编码器和定位系统高精度

**仿真数据 (C001)**:
- 人工模拟生成
- 几何参数可能存在**建模误差**
- 角度标签可能有**单位转换错误**（度 vs 弧度）
- **Pitch值的符号可能不正确**（我们发现是负值）
- 探测器像素映射可能有**微小偏差**

### 2. 数据质量和一致性

**真实数据**:
```
投影数量: 48590（完整扫描）或 2000（子集）
探测器: 736×64（标准临床配置）
视图分布: 均匀、连续
角度步长: 由扫描仪控制，极其规律
```

**仿真数据**:
```
投影数量: 约2000个（可能下采样）
探测器: 736×64
视图分布: 可能有间隙或不规律
角度步长: 计算得出，可能有累积误差
```

### 3. Katsevich算法的敏感性

Katsevich算法对**几何精度**要求极高：
- 角度误差 >0.1° 会导致严重伪影
- Pitch值误差 >1% 会产生螺旋伪影
- 探测器中心偏移 >0.5像素会导致模糊

**真实数据**: 硬件保证几何精度
**仿真数据**: 可能存在建模误差累积

### 4. 具体问题推测

仿真数据C001可能的问题：

#### 问题1: 角度符号错误
```python
# 真实数据: 角度由扫描仪记录，符号正确
真实角度: [0, -0.011, -0.022, ...] rad  # 逆时针

# 仿真数据: 可能角度方向反了或pitch符号错
仿真角度: [0, -0.011, -0.022, ...] rad  # 但实际应该是正的？
```

#### 问题2: Pitch定义不一致
```
真实数据: Pitch = table_feed / 360° (标准定义)
仿真数据: Pitch可能是 = table_feed / 2π (弧度定义)
         或者符号反了（我们看到-3.67 mm/rad）
```

#### 问题3: 探测器坐标系
```
真实数据: U-V坐标正确映射到物理探测器
仿真数据: 可能转置后没有正确调整PixelSpacing（我们修复了这个）
```
